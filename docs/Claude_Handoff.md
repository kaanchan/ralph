## 1. What RALPH Is

RALPH is a local-first recursive coding agent. Given a task description and a target git repo, it:

1. **Plans** — LLM generates an implementation plan
2. **Executes** — LLM writes `solution.py` and `test_solution.py` to the target repo
3. **Evaluates** — runs pytest, scores the result
4. **Loops** — retries up to N times; escalates to a cloud model after N failures
5. **Commits** — git commits any successful edits

The key design goal: **use a local quantized model (qwen25-coder-14b via Ollama) for cost-free iterations, only escalating to cloud (Gemini) when local fails repeatedly.**

Stack: Python 3.12, LangGraph (graph orchestration), LiteLLM (cloud model calls), Aider (cloud model code editing), direct Ollama HTTP API (local model code generation), Rich (console output).

## 2. The Core Problem Being Solved

**Local quantized models cannot follow Aider's structured edit formats.**

Aider requires models to output either SEARCH/REPLACE blocks, udiff hunks, or complete whole-file rewrites in a specific format. `qwen25-coder-14b` (and similar 14b parameter quantized models) consistently ignores format instructions and outputs whatever its training data shaped it toward — usually markdown code blocks without the surrounding structure Aider expects.

**Symptoms when using Aider with a local model:**
- Aider reports "no edit applied" even though model responded
- File remains at placeholder content (`# generated by ralph`)
- Aider sometimes creates a phantom file called "Assistant" with the model's response
- Solution: bypass Aider entirely for local models; call Ollama `/api/chat` directly

### executor.py — Local vs Cloud Split

```
executor_node()
  └─ run_aider(task, model, repo_dir)
       ├─ _write_file(sol_prompt, model, solution_path, ...)
       │    ├─ if model.startswith("ollama/") → _write_file_local()
       │    │    └─ _call_ollama()  ← direct HTTP to Ollama /api/chat

**Cloud path:** Aider subprocess with `--model gemini/gemini-2.5-flash --edit-format diff-fenced`. Streaming reader thread captures output. Aider handles multi-file context and SEARCH/REPLACE correctly for cloud-scale models.

## 5. The Remaining Unsolved Problem

### Symptom

When RALPH runs with the local model, `solution.py` is generated correctly (~3s). Then `test_solution.py` **times out** at 90s (the previous timeout) or triggers resource contention on Ollama for subsequent calls.

```
[23:09:07] local -> solution.py   finished in 3.0s ✓
[23:09:10] local -> test_solution.py  ...
[23:10:40] *** RALPH TIMEOUT — local model test_solution.py  (90s budget exhausted)
```

### What We Know About the Timeout

**Direct Ollama tests without the RALPH plan included:**
- `test_solution.py` prompt (short, ~200 chars): completes in **4.9–10s** ✓

**RALPH executor with a 654-char plan in the prompt:**
- `test_solution.py`: completes in **4.7s** ✓ (one successful run observed)

**RALPH executor with a 1108-char plan in the prompt:**
- `test_solution.py`: **90s timeout every time** ✗

**Key observation:** The plan is prepended to every prompt. The `executor_node` builds:
```python
task_prompt = f"{state['plan']}\n\nTask: {state['task']}"
```
Both `sol_prompt` and `test_prompt` include this full `task_prompt`. The plan is ~1108 chars in typical runs. `solution.py` completes fine with this same plan. `test_solution.py` does not.

**Why test_solution.py is slower than solution.py:**  
The test file generation prompt asks for more complex output (multiple test functions, imports, assertions). The model may generate 3–5x more tokens for a test file than a 2-line solution. At ~30 tok/s for a 14b model on this hardware, generating 200+ lines of tests could genuinely take 90–180s.

### The Socket/Cancellation Problem (Compounding Issue)

When the `LOCAL_MODEL_TIMEOUT` fires (either via `concurrent.futures.TimeoutError` in the old code or `TimeoutError` raised inside the streaming loop in the new code), **Ollama continues generating on the server side**. The model is loaded in VRAM and the HTTP request is still active from Ollama's perspective.

The next RALPH call (planner) then hits Ollama immediately and finds it busy — LiteLLM's 30s planner timeout fires, crashing the whole run.

`ollama ps` confirms: after a cancelled test_solution.py call, model shows `Stopping...` for 60–90+ seconds before Ollama becomes available again.

**This is the core unsolved loop:**
1. test_solution.py times out → we cancel our read
2. Ollama keeps generating → blocks next call
3. Next run's planner times out → crash
4. Repeat

## 7. Unresolved Questions

### 7a. Does Ollama support request cancellation?

**Unknown.** Ollama's REST API (`/api/chat`) does not document an abort/cancel endpoint. A client disconnect (closing the socket) does NOT stop server-side generation — tested empirically. This is the root of the resource contention problem.

**Possible approaches:**
- `POST /api/cancel` or `DELETE /api/generate/{id}` — check Ollama docs/source; may exist in recent versions
- Ollama's `keep_alive` parameter: `"keep_alive": 0` forces model unload after request. Could be used post-generation but doesn't help mid-generation.
- External process management: `taskkill /F /IM ollama.exe` between calls — destructive, restarts the server
- Accept the wait: if test generation takes 120s, budget 180s, accept the sequential nature

### 7b. Is the 90–180s generation time expected?

Plausible. `qwen25-coder-14b` on an RTX 3090 at ~30 tok/s. A comprehensive test file with 15 test functions, imports, docstrings could be 200+ tokens. 200 tok / 30 tok/s ≈ 7s. But with prefill of a 1000-char plan (~250 tokens input), total could be 15–30s. Still not 90s+.

**Unexplained:** why does 654-char plan → 4.7s but 1108-char plan → 90s+? This is a ~2x prompt increase causing a 20x time increase. Something non-linear is happening. The model may be entering a self-reflection or chain-of-thought mode with the longer plan, generating internal reasoning before the code block.

**Recommendation:** Capture the raw streaming output during a long run (log every token) to see what the model is actually generating before the timeout fires.

### 7c. Should the plan be included in the test prompt?

The plan is implementation-focused (function signatures, types, success criteria). The test generator needs to know the function name and expected behavior — the full implementation plan adds noise. **Removing the plan from the test prompt** would shorten it significantly and likely fix the timeout.

The simplest experiment: change `test_prompt` to use only `state['task']` not `task_prompt` (plan + task).

    """Block until Ollama is not busy, or max_wait seconds pass."""

        try:
            # Ollama has no formal "busy" flag; use a cheap streaming request
            # that completes fast and observe if it gets queued
            ...
        except:
            pass
        time.sleep(3)
```

Or more practically: after a timeout, add an `ollama ps`-equivalent check before retrying.
